CHAPTER 1: PROJECT INTRODUCTION
1.1 Executive Summary
The Street Children Fingerprint Management System is a web-based biometric identification and registration platform designed to support the registration, identification, and management of street children. The system enables healthcare workers and authorized staff to capture child demographic information, enroll fingerprints, verify identities, manage users, and operate in both online and offline environments.
The platform improves record accuracy, prevents duplicate registrations, and provides a secure method of identifying children through biometric fingerprint technology.
1.2 Business Problem
Organizations working with vulnerable children often face challenges including:
    • Duplicate child registrations.
    • Loss of identification records.
    • Inability to accurately identify returning children.
    • Manual data collection processes.
    • Poor tracking of service delivery.
This system addresses these challenges by introducing biometric fingerprint identification, centralized child records, role-based access control, and offline synchronization capabilities.
1.3 SYSTEM OBJECTIVES
The system aims to:
• Uniquely identify street children using biometrics.
• Reduce duplicate registrations.
• Improve service delivery tracking.
• Support offline field operations.
• Provide secure access to sensitive information.
• Enable reporting and analytics.
1.4 Project Scope
In Scope
    • Staff Authentication
    • User and Role Management
    • Patient Registration
    • Fingerprint Enrollment
    • Fingerprint Verification
    • Dashboard Analytics
    • Notifications
    • Audit Tracking
    • Offline Data Collection
    • Data Synchronization
    • Outreach Location Management
Out of Scope
    • Payroll Management
    • Financial Accounting
    • Mobile Native Applications
    • SMS Gateway Integration
    • National ID Integration
    • Electronic Medical Records (EMR)

2.1 High-Level Architecture
The Street Children Fingerprint Management System follows a multi-tier client-server architecture. The architecture separates presentation, business logic, and data storage layers to improve maintainability, scalability, and security.
Users interact with the web application through a browser. Requests are processed by the back-end API, which communicates with the database and biometric services. Fingerprint devices communicate with the biometric module through the installed SDK.

CHAPTER 4: USER ROLES & ACCESS CONTROL
4.1 Overview
The Street Children Fingerprint Management System uses a kind of security called Role-Based Access Control. This means that people who use the system can only do things that are part of their job.
The Street Children Fingerprint Management Systems Role-Based Access Control makes the system more secure. It keeps information and fingerprints safe. It also stops people from getting to parts of the system when they should not be able to. Each person who uses the system gets a role. Each role can do things and not others.
This way of doing things makes it easier to manage who can do what. The Street Children Fingerprint Management Systems Role-Based Access Control also helps make sure that people follow the rules and that patient information is protected.

CHAPTER 5: DATA MODEL
5.1 Overview
The Street Children Fingerprint Management System is a system that uses a kind of database to keep all the information safe and organized. This database is like a container that holds all the important information, about the patients, including their fingerprints and other personal details. It also keeps track of who's using the system what they can do and what is happening in the system.
The database is designed to make sure that all the information is safe and can be retrieved easily. It helps to connect all the parts of the system like the patients, the people who work with them and the people who manage the system.
The Street Children Fingerprint Management System database is made using MySQL, which is a type of database software. It is set up in a way that reduces repetition of information and makes it easier to maintain.
5.2 Entity Relationship Diagram (ERD)
The Entity Relationship Diagram is a diagram that shows how all the different parts of the Street Children Fingerprint Management System are connected.
The Entity Relationship Diagram shows how the users, the roles they play what they are allowed to do the patients, the fingerprints, the locations, the notifications and the audit logs all work together in the Street Children Fingerprint Management System.

Figure 5.1: Entity Relationship Diagram
5.3 Core Entities
Staff Users
The Staff Users entity stores information about all authorized system users.
This entity manages:
    • User authentication credentials
    • Personal profile information
    • Assigned roles
    • Account status
    • User activities
Examples of staff users include:
    • System Administrators
    • Nurses
    • Doctors
    • Lab Technicians
    • Pharmacists
Roles
The Roles entity defines user responsibilities and access levels within the system.
Examples include:
    • Super User
    • Nurse
    • Doctor
    • Lab Technician
    • Pharmacist
Each role is associated with a set of permissions that determine what actions a user can perform within the application.
Permissions
The Permissions entity defines individual system privileges.
Examples include:
    • Create Patient
    • Update Patient
    • Delete Patient
    • Enroll Fingerprint
    • Verify Identity
    • Manage Users
    • Create Notifications
Permissions are assigned to roles through the Role-Based Access Control (RBAC) framework.
Patients
The Patients entity stores demographic and registration information for all registered individuals.
Information stored includes:
    • Unique patient identifier
    • Full name
    • Gender
    • Estimated birth year
    • Photographs
    • Registration location
    • Registration date
    • Responsible staff member
This entity serves as the primary record for each beneficiary registered in the system.
Fingerprints
The Fingerprints entity stores biometric fingerprint templates associated with registered patients.
Information stored includes:
    • Finger index
    • Biometric template
    • Quality score
    • Verification status
    • Enrollment date
A single patient may have multiple fingerprint records.
Locations
The Locations entity stores information about outreach centers, registration facilities, and service delivery points.
This enables the organization to track where registrations and biometric activities occur.
Notifications
The Notifications entity manages system alerts, announcements, reminders, and informational messages distributed to users.
Notifications can be targeted to:
    • All users
    • Specific roles
    • Individual users
Audit Logs
The Audit Logs entity maintains a record of critical system activities for accountability and compliance purposes.
Examples include:
    • User logins
    • Patient registration
    • Fingerprint enrollment
    • Record modifications
    • Permission changes
    • Synchronization events
Audit logs help ensure transparency and traceability within the system.

5.7 Data Integrity and Security
To ensure reliability and security, the database implements the following controls:
    • Primary Keys
    • Foreign Keys
    • Referential Integrity Constraints
    • Unique Constraints
    • Password Hashing

CHAPTER 7: OFFLINE SYNCHRONIZATION
7.1 Overview
The Street Children Fingerprint Management System is designed to operate effectively in environments where internet connectivity may be unreliable or unavailable. To support field operations, the system includes an Offline Synchronization Engine that enables staff to continue registering patients, capturing biometric fingerprints, and updating records while disconnected from the central server.
All changes performed during offline operation are temporarily stored in the local database and automatically synchronized with the central server once network connectivity becomes available.
This functionality ensures uninterrupted service delivery and minimizes the risk of data loss during outreach activities.
7.2 Offline Operation Architecture
The synchronization architecture consists of four primary components:
Component
Description
User Interface
Allows staff to continue working while offline
Local Database
Stores records created or modified offline
Sync Worker
Detects connectivity changes and manages synchronization
Central Server
Stores master records and resolves synchronization conflicts

7.3 Synchronization Process
When internet connectivity is unavailable, the system automatically switches to offline mode. Users can continue performing authorized operations without interruption.
Once connectivity is restored, the synchronization service automatically begins transferring locally stored records to the central server and retrieving any new updates.
Synchronization Steps
    1. User works in offline mode.
    2. Data is stored in the local database.
    3. Connectivity becomes available.
    4. Sync Worker detects the connection.
    5. Pending local changes are uploaded.
    6. Updated records are downloaded from the server.
    7. Conflict resolution is performed if necessary.
    8. Local database is updated.
    9. Synchronization status is confirmed.
7.4 Offline Synchronization Workflow
The Offline Synchronization Workflow illustrates how the system continues to operate when internet connectivity is unavailable and how data is synchronized once connectivity is restored. Field staff continue using the application normally even when an internet connection is unavailable. During this period, users can register patients, capture fingerprints, update records, and perform other authorized activities without interruption. All data created or modified while offline is stored securely within the local database on the user's device. This ensures that no information is lost due to network unavailability.

7.6 Conflict Resolution
Data conflicts may occur when the same record is modified both locally and on the server before synchronization.
The system resolves conflicts using predefined synchronization rules.
Examples include:
    • Latest modification timestamp wins.
    • Server record takes precedence.
    • Manual review required for critical records.
All synchronization conflicts are logged for audit purposes.
7.7 Benefits of Offline Synchronization
The Offline Synchronization Engine provides several operational benefits:
    • Continuous operation without internet connectivity.
    • Reduced risk of data loss.
    • Improved reliability during field outreach activities.
    • Automatic synchronization once connectivity is restored.
    • Reduced workload for field staff.
    • Consistent data across all devices and locations.
    • Supports reliable patient registration and biometric identification in remote environments.
    • Enables uninterrupted service delivery in areas with poor or no internet connectivity.
